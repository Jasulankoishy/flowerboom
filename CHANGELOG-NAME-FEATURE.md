# Добавлена функция запроса имени при первом входе

## Дата: 24 апреля 2026

## Что добавлено

### Бэкенд

1. **Новый endpoint `POST /api/auth/set-name`**
   - Защищён JWT токеном
   - Валидация: 2-50 символов
   - Обновляет поле `name` в базе данных

2. **Флаг `isNewUser` в ответах авторизации**
   - Добавлен в `register()` - возвращает `true` если `name` пустое
   - Добавлен в `login()` - возвращает `true` если `name` пустое
   - Добавлен в Google OAuth callback - передаётся через URL параметр

3. **Функция `setName()` в authController**
   - Принимает имя пользователя
   - Валидирует длину (2-50 символов)
   - Обновляет запись в БД

### Фронтенд

1. **Новый шаг "name" в AuthPage**
   - Показывается после успешной регистрации/входа если `isNewUser === true`
   - Форма с полем ввода имени
   - Валидация на клиенте
   - Тёмный дизайн в стиле сайта

2. **Обновлён authStore**
   - Добавлен метод `setName(name: string)`
   - Обновляет имя пользователя в сторе

3. **Обновлён ProfilePage**
   - Показывает имя пользователя (или email если имени нет)
   - Кнопка редактирования имени
   - Инлайн форма для изменения имени
   - Кнопки "Сохранить" и "Отмена"

4. **Обновлён AuthCallbackPage**
   - Обрабатывает параметр `isNewUser` из Google OAuth
   - Перенаправляет на форму имени если нужно

5. **Обновлён API**
   - Добавлен endpoint `setName` в config.ts
   - Добавлен метод `setName()` в auth.ts
   - Добавлен флаг `isNewUser` в `AuthResponse`

## Как это работает

### Сценарий 1: Регистрация через Email+Password

1. Пользователь регистрируется
2. Сервер создаёт пользователя с `name: null`
3. Сервер возвращает `isNewUser: true`
4. Фронтенд показывает форму ввода имени
5. Пользователь вводит имя
6. Имя сохраняется через `POST /api/auth/set-name`
7. Перенаправление на главную

### Сценарий 2: Вход через Email+Password (существующий пользователь)

1. Пользователь входит
2. Сервер проверяет поле `name`
3. Если `name` пустое → `isNewUser: true` → форма имени
4. Если `name` заполнено → `isNewUser: false` → главная страница

### Сценарий 3: Google OAuth

1. Пользователь входит через Google
2. Passport.js создаёт/обновляет пользователя
3. Google предоставляет `displayName` → сохраняется в `name`
4. Callback проверяет `!user.name`
5. Если имени нет → `isNewUser=true` в URL → форма имени
6. Если имя есть → сразу на главную

### Сценарий 4: Изменение имени в профиле

1. Пользователь открывает `/profile`
2. Видит своё имя (или email)
3. Нажимает кнопку редактирования
4. Вводит новое имя
5. Нажимает "Сохранить"
6. Имя обновляется через `POST /api/auth/set-name`

## Технические детали

### Валидация имени

**Клиент:**
- Минимум 2 символа
- Максимум 50 символов

**Сервер:**
- Минимум 2 символа (после trim)
- Максимум 50 символов (после trim)
- Автоматический trim при сохранении

### Безопасность

- Endpoint защищён middleware `authenticateToken`
- JWT токен передаётся в заголовке `Authorization: Bearer <token>`
- Пользователь может изменить только своё имя (берётся из `req.user.userId`)

### База данных

Поле `name` уже существовало в схеме:
```prisma
model User {
  name String? // необязательное
}
```

Миграция не требуется.
